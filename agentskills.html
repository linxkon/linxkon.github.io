Agent与Skills

Agent 这玩意儿，光聪明没用

我最开始搞 Agent 的时候，总觉得是模型不够强。后来发现，根本不是智商问题。它就像个刚毕业的数学天才，让他去报税，他可能连“专项附加扣除”是什么都不知道，更别说那些藏在公司历史邮件里的奇葩约定了。

Anthropic 那篇讲 Skills 的文章，最戳我的就一句话：The new paradigm: code is all you need。

这话听着像废话，但背后意思很硬：代码正在变成 Agent 干数字活的统一接口。以前我们老想着给 Agent 定制各种花里胡哨的工具，现在思路变了——给它一个代码执行环境和文件系统，让它自己用代码去搭脚手架。

“代码就是一切”，工程上到底怎么搞？

我们习惯把“写代码”当成一个具体任务。但 Anthropic 说得更狠：在 Agent 的世界里，代码是操作系统层的接口。

举个例子，让 Agent “生成一份财务报告”。传统做法是啥？写一堆工具说明，塞一堆上下文，然后祈祷它每次输出都差不多。结果呢？十次能有八次格式不一样，气得你想砸键盘。

“code is all you need”的路子，是把任务拆成确定的执行步骤： 1. 调 API 拉数据，扔到文件系统里。 2. 用写好的脚本清洗、统计。 3. 用模板生成报告骨架。 4. 最后，才让模型去做它擅长的部分：综合判断，组织叙事。

这么一来，Agent 的能力不靠“重造轮子”，就看它“会不会用代码把事串起来”。Bash + 文件系统 + 一堆可复用脚本，成了最实在的脚手架。

通用 Agent 为什么还是难用？缺的不是脑子，是经验

文章里那个比喻太对了：报税你找数学天才还是老会计？肯定选老会计。为什么？经验。

现在的通用 Agent 就卡在这儿。它推理能力不差，但一干真活就露怯： * 背景缺失：公司里的潜规则、历史决策、合规红线，它默认不知道。你每次都得从头解释，累不累？ * 复用困难：同一个任务，这次输出挺好，下次就飘了。质量全靠人盯着，根本没法规模化。 * 不会长记性：这次踩过的坑，下次一模一样再踩一遍。因为“经验”没沉淀下来，全散在对话历史里，跟没说一样。

Skills 要补的，就是这块“职业训练”。它不是让 Agent 更聪明，是让它更专业。

Skills 本体：把 SOP 做成能版本控制的文件

Anthropic 对 Skills 的定义非常工程，不是什么玄学概念。它就是一组文件的组织方式，用来打包领域知识和流程。

关键不是目录多漂亮，而是它带来的工程属性： * 能版本控制：能用 Git 管，能 Code Review，能回滚，出事了能追责。 * 能共享：丢到团队仓库里，谁都能用。 * 能审计：流程和边界白纸黑字写在文件里，不是藏在某次聊天记录里。

说白了，Skills 就是把“教新人干活”的那套材料，做成了可组合、可复用的资产包。

最核心的：渐进式披露，别一次性喂吐它

目录结构只是表象。真正让 Skills 能规模化而不崩掉的，是渐进式披露（Progressive Disclosure）——按需加载，别一次性把几百页文档全塞进上下文。

他们搞了个三层“账本”： 1. 元信息（metadata）：只暴露 name 和 description，大概 50 个 token，用来匹配和发现技能。 2. 入口（SKILL.md）：匹配上了才加载，大概 500 个 token，说清楚流程、边界和验收标准。 3. 附录（references/）：需要细节的时候再按文件名去读，单个文件可能 2000+ token，但不常驻内存。

这个设计救了我老命。以前动不动就上下文爆炸，现在清爽多了。

几个踩过的坑，和一点不确定

Description 不是简介，是路由规则 很多团队（包括我一开始）最容易把 description 写废。它不该是技能简介，而是一条路由规则。要写清楚两件事：1）什么情况下该用这个技能；2）用了之后会产出什么具体东西。 我的经验是，用你们团队真实的黑话和触发词来写。别整那些抽象的“分析能力”、“处理功能”。

脚本就是工具，把确定性工作锁死 文章里有个例子我印象很深：他们发现 Claude 老是写同一段脚本，用来给演示文稿套 Anthropic 的模板。每次都现场重写，既浪费 token 又容易出错。 后来他们干脆把脚本存下来，让 Agent 直接调用。收益太明显了： * 确定性拉满：关键逻辑走脚本，失败了好归因。 * 好维护：脚本能迭代、能 Review、能补边界条件。 * 上下文干净：脚本本体不用进上下文，只把运行结果（输出或生成的文件）带回来就行。

我现在把模型定位成“编排者”，脚本定位成“操作工”。很多玄学的稳定性问题，一下子变成了可调试的工程问题。

Skill、MCP、Subagent 别搞混 这几个概念特别容易揉成一团，最后变成“一个超级 Prompt + 一堆工具”，系统很快就失控了。 我的分工理解是： * Skills：管“怎么做”（流程、模板、检查点）。 * MCP：管“连到哪”（外部工具和数据源的连接层）。 * Subagents：管“怎么分”（并行、审计、隔离上下文）。

你可以把它们看成三种不同的“工程杠杆”：方法、连接、组织。别用错地方。

团队怎么落地？从1个Skill开始

别想着一口吃成胖子。我的建议是：从1个能反复用、能验收的Skill开始。

第一步：选对题 就看两个信号： 1. 高频：每周或者每个项目都要做的。 2. 可验收：输出有明确标准，不靠“感觉”。 比如：PR Review、故障排查Checklist、周报生成、竞品信息拉取对比。这些都很适合开局。

第二步：写对入口文件 SKILL.md 别写成百科全书。就写三件事： 1. 流程：第一步干啥，第二步干啥。 2. 边界：什么情况下别动手，缺什么信息要先问。 3. 验收：怎么算做完？（最好能机器检查） 那些细节、例子、数据口径表，全拆到 references/ 文件夹里，用的时候再读。

第三步：能脚本化的，绝不让模型即兴发挥 模型擅长处理“不确定”，比如拆解问题、权衡取舍、组织语言。但像数据清洗、固定格式生成、模板套用这些确定性的脏活累活，必须用脚本锁死。别考验模型的耐心，更别考验你的。

第四步：Skill多了，就得治理 技能少的时候，靠人脑记得住。超过20个，就必须上治理了。 最简单的治理： * 每个 Skill 标清楚 Owner。 * Description 写明白触发场景，避免技能之间打架。 * 定期盘点：合并重复的，删掉没人用的，补齐验收标准。

我现在也不确定的是，当 Skills 生态真的跑起来，跨团队、跨公司共享的时候，版本兼容性和质量评估到底该怎么搞？会不会出现“垃圾技能”污染整个池子？这玩意儿，可能得踩过坑才知道。
