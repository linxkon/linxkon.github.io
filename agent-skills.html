Agent Skills

我为什么放弃了Agent Skills

我一开始对Claude的Agent Skills特别上头。这玩意儿听起来太对了——把那些我每天要跟Claude重复八百遍的指令、工作流、最佳实践，打包成一个“技能”，让AI自己判断什么时候该用。这不就是解决“AI失忆”和“提示词工程疲劳”的终极方案吗？我脑子里立刻蹦出一堆应用场景：代码评审规范、内部API文档查询、甚至是我们团队那套复杂的部署流程。

但用了一阵子，我发现它跟我想的完全不是一回事。问题就出在这个“自动判断”上。

所谓的“智能触发”根本靠不住

官方说Claude会在“相关时”自动使用Skills。这个“相关”的判断标准，对我来说就是个黑盒。我写了个Skill，专门用来按照我们公司的代码规范格式化Python文档字符串。结果呢？我直接问“给这个函数加个docstring”，Claude十次里有六七次根本不会触发那个Skill。它要么自己瞎编一个格式，要么就简单写两句。

我得在问题里非常明确地提到“用我们公司的Python文档规范”，或者甚至把Skill的名字打出来，它才好像“哦，你想用那个啊”，然后不情不愿地去读SKILL.md。那这跟我直接把规范贴进对话里有什么区别？还多了一层不可靠的“触发”逻辑。所谓的“自动化”成了“薛定谔的自动化”，用起来心里完全没底，每次都得猜它这次会不会听话。

文件系统访问是个伪命题

Skills宣传的一个亮点是能访问VM里的文件系统，可以捆绑脚本、模板这些资源。听起来很强大对吧？我试了。

我打包了一个小脚本和一个JSON模板文件进去，想着让Claude能动态生成一些配置。结果操作过程极其反直觉。Claude不会“主动”去读这些附加文件，它需要你在Skill的指令里，用明确的bash命令告诉它去读取、去执行。比如，你得在SKILL.md里写：“现在，请运行 cat ./resources/template.json 来获取模板”。

这感觉就特别割裂。我是在跟一个“智能体”对话，但我指挥它干活的方式，却像是在写一个自动化shell脚本。而且，一旦涉及到稍微复杂点的逻辑，比如条件判断或者循环处理多个文件，写在Markdown里的指令就会变得又臭又长，难以维护。有这功夫，我为什么不直接写个Python脚本，然后让Claude去调用这个脚本的接口？后者反而更清晰、更可控。

预构建的技能？聊胜于无

Anthropic自己提供的那几个预构建Skill，像处理PPT、Excel、Word、PDF的，我大概试了试。生成一个简单的表格或者格式化一个文档，勉强能用。但一旦需求稍微复杂点，比如“分析这个Excel里的数据趋势并生成带图表和洞察的报告”，它生成的东西就非常表面，根本撑不住真实的业务场景。

这些预置技能更像是一个“能力声明”，告诉你Claude理论上能碰这些文件格式了。但真要靠它完成专业工作，里头的指令和逻辑还差得远。它解决的是“从0到0.5”的问题，但用户需要的是“从0.5到10”的深度解决方案。

我最终的选择：回归精准的上下文

所以折腾了一圈，我后来基本放弃了用Skills来管理复杂能力。对于那种需要严格遵循、步骤固定的工作流，我现在的做法更简单粗暴：

  1. 制作超级提示词模板：把所有的指令、规范、例子，做成一个结构清晰的Markdown模板，保存在我的笔记软件里。
  2. 需要时，精准粘贴：开始一个新对话，或者在一个关键节点，我把整个模板作为第一条消息或者一条系统指令发过去。虽然会占用不少tokens，但换来的是百分之百的确定性。我知道Claude这次看到了全部规则，没有遗漏。

Skills那个“渐进式披露”的设计（先加载元数据，触发再加载指令，按需加载资源），本意是省token。但对我来说，可靠性远比那点token开销重要。我宁愿多花点钱，也要确保AI拿到了完整的“任务说明书”，而不是一个可能被它忽略的“目录”。

我现在也不确定，这种“让AI自己判断该用什么技能”的路径是不是从根本上就有问题。也许对于极其简单、边界清晰的任务（比如“把这句话翻译成法语”）是有效的。但一旦任务变得复杂、模糊，需要人类式的理解和判断时，这种基于关键词匹配的“自动触发”机制就显得无比笨拙。

最后留个问题：如果“自动触发”不靠谱，那Skills剩下的核心价值，是不是就只是一个“可复用的提示词打包格式”？如果真是这样，我们为什么需要这么一个复杂的、基于文件系统的架构，而不是一个更简单的、在对话中就能管理和插入的“提示词片段”库？
